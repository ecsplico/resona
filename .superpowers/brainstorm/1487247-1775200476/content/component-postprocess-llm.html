Why litellm?

Unified API

  • OpenAI, Anthropic, Mistral, Cohere, etc.
  • Local: Ollama, vLLM, llama.cpp, LM Studio
  • One function call: litellm.completion()

Env Var Config

  • OPENAI_API_KEY — for OpenAI/compatible
  • ANTHROPIC_API_KEY — for Claude
  • Model string selects provider:
    ollama/llama3, gpt-4o, claude-sonnet-4-20250514

LLM Postprocessing Step

resona_postprocess/llm.py
import litellm
from decouple import config

def llm_postprocess(
    text: str,
    *,
    prompt: str,
    model: str | None = None,
    api_base: str | None = None,
) -> str:
    """Send transcript through an LLM with a given prompt."""
    model = model or config("RESONA_LLM_MODEL", default="gpt-4o-mini")
    api_base = api_base or config("RESONA_LLM_API_BASE", default="") or None

    response = litellm.completion(
        model=model,
        api_base=api_base,
        messages=[
            {"role": "system", "content": prompt},
            {"role": "user", "content": text},
        ],
    )
    return response.choices[0].message.content

Pipeline with LLM Steps

Composable: static replacements → LLM → more steps
# Example: build a pipeline with both static and LLM steps

pipeline = PostprocessPipeline()

# Step 1: static replacements (fast, local)
pipeline.add(
    "replacements",
    lambda text: apply_replacements(text, replacements),
)

# Step 2: LLM formatting (configurable prompt)
pipeline.add(
    "llm-format",
    lambda text: llm_postprocess(
        text,
        prompt="Format this medical transcription with proper "
               "paragraphs, headings, and punctuation. "
               "Do not change the content, only the formatting.",
    ),
)

# Run the full chain
md = pipeline.run(raw_text)

Each step runs in order. LLM steps can override model per-step or fall back to RESONA_LLM_MODEL.

How sources.py Builds the Pipeline

sources.py — updated
def build_pipeline_from_config(
    config_path: Path | None = None,
) -> PostprocessPipeline:
    """Build pipeline from ~/.resona/postprocess.json"""
    config_path = config_path or Path.home() / ".resona" / "postprocess.json"
    if not config_path.exists():
        return PostprocessPipeline()  # no-op pipeline

    cfg = json.loads(config_path.read_text())
    pipeline = PostprocessPipeline()

    for step in cfg.get("steps", []):
        if step["type"] == "replacements":
            rules = load_replacements_from_file(step.get("source"))
            pipeline.add("replacements", lambda t: apply_replacements(t, rules))

        elif step["type"] == "llm":
            prompt = step["prompt"]
            model = step.get("model")  # None → falls back to env var
            pipeline.add(
                step.get("name", "llm"),
                lambda t, p=prompt, m=model: llm_postprocess(t, prompt=p, model=m),
            )

    return pipeline

Server-Side (resona-api) — Same Concept

The API builds its pipeline from DB records instead of a JSON file, but uses the same PostprocessPipeline and llm_postprocess(). LLM prompts become a new DB table (like InitialPrompt today but for postprocessing), manageable via API endpoints and CLI commands.

This means resona prompts evolves: initial prompts (pre-transcription hints) stay separate from postprocess prompts (LLM instructions applied after).
